PLB-SPEC-001 · v1.1 · Based on UX audit v1 + review pass

PromptLab Redesign

This spec translates the UX audit into buildable screen descriptions, change tickets, and rollout sequencing. Every section maps to a concrete area of App.jsx, options.html, options.js, or background.js.

v1.1 changes

Panel governance rule added (Section 07). Library drawer expanded with Insert/Preview/Open actions. History save behavior clarified — auto-save run metadata, explicit save required for winner and rationale. Accessibility moved to a per-phase release gate. Composer extracted into its own section (06) with a defined long-term home. Status chip model names updated to reflect actual current models.

Core thesis

PromptLab has the right feature set. The redesign is not about adding — it's about reducing decision cost at every step. One primary action per screen. Progressive disclosure over control dumps. Unified mental models over fragmented surfaces.

Proposed structure

Three primary workspaces replace five equal tabs. Pad moves to a utility overlay or bottom sheet. Settings becomes a dedicated admin surface.

Create
Default landing. Prompt input, enhance, compare, save.
Library
Saved prompts. Browse, use, manage.
Experiments
A/B, history, outcome logging.
Pad (overlay)
Scratch space. Accessible from any screen.
Settings (admin)
Connection Center + preferences.

PLB-SPEC-001 · Section 02

Create screen

Default landing. Single primary action: Enhance. Everything else is progressive — only surfaces when relevant.

Primary CTA

One Enhance button, full width, bottom of input area. No action competes with it visually on this screen.

Workflow stages

Linear progression. Each stage reveals only what is needed. No skipping required — stages can be collapsed/hidden if unused.

1

Write

Prompt textarea (primary). Model mode selector (inline below textarea, compact toggle). Checklist chip strip (optional, collapsed by default). No lint display yet.

2

Review suggestions

Lint and quality scoring surface after the user pauses or manually triggers. Presented as dismissible chips above the Enhance button, not a stacked column. Scores shown as a single composite number, not a breakdown grid, until the user expands.

3

Generate

Enhance button fires. Output area appears below the input, separated by a divider. Loading state is an inline skeleton in the output zone, not a spinner overlay.

4

Compare

Output area has four tabs: Improved · Diff · Variants · Notes. Default tab is Improved. Tabs are rendered at the top of the output zone only — no vertical scrolling through four separate blocks.

5

Save

Fast save: a "Save" button in the output toolbar saves immediately with an auto-generated title (first 8 words of the prompt) and no drawer. A confirmation chip appears inline for 3 seconds with an "Edit details" link.
Full save: "Edit details" or a dedicated secondary affordance next to Save opens the right-side drawer with title field, tags, collection picker, and notes. Use a split button or chevron menu on desktop/web; do not rely on long-press as the only way to reach full-save behavior. The drawer is non-blocking — the editor remains usable behind it. One-panel-at-a-time: if the Library drawer is open, Save replaces it.

Layout zones

Editor toolbar: model toggle · checklist · quality score chip · [Library →]
Prompt input (Stage 1)
Textarea, full width, resizable. No decorations inside the box.
Lint chips (Stage 2) — collapsed until content present
[ Enhance ] ← single CTA, full width
Output area (Stage 3+) — hidden until first run
Tabs: Improved · Diff · Variants · Notes
Output toolbar: Copy · Send to A/B · Save ↗

Quick inject

Quick inject stays but moves to a collapsible strip at the top of the input area, triggered by a "+" button. It does not stack below the output. Source: App.jsx:1453

Library drawer (from Create)

Triggered by the "Library →" button in the editor toolbar. Opens as a right-side overlay panel, 320px wide. The panel shows a search field and a flat list of saved prompts. For reuse-heavy users, the drawer must support three actions per prompt — not just load-and-close:

ActionBehavior
InsertAppends prompt text at cursor position without replacing current content. Drawer stays open.
LoadReplaces input with selected prompt. Drawer closes.
PreviewShows full prompt text in an inline expansion within the drawer, no nav change.
Open in Library ↗Navigates to Library screen with this prompt focused. Loses current Create context — show a discard warning if input has content.

One-panel-at-a-time rule applies: if the Save drawer is open, opening the Library drawer closes Save first. See Section 07.

PLB-SPEC-001 · Section 03

Library screen

Browsing and management surface for saved prompts. Primary action: Use prompt. All destructive and administrative actions moved to an overflow menu or detail panel.

Card action audit

Current cards carry: Copy, Load, Compose, Share, Edit, Rename, Expand, Reorder, Restore, Delete. Source: App.jsx:1338. That is 10 actions per row. Proposed: 2 visible (Use, Copy) + 1 overflow (More ···).

Detail panel

Clicking a card's title or hitting Enter on a focused card opens a right-side panel (not an expanding card row). The panel is 380px wide and contains full prompt text, edit controls, version history, usage stats, and notes. The list behind it does not reflow — no list jitter. Source fix: App.jsx:1338 expand behavior.

List controls

Search field (top, full width). Filter chips below: All · Recent · Favorites · by Collection. Sort dropdown (right side). No inline reorder drag by default — Move up / Move down available in the overflow menu. Drag-and-drop is an opt-in power user feature, not the primary reorder mechanism.

PLB-SPEC-001 · Section 04

Experiments screen

A/B testing and prompt history, connected. Primary action: Run comparison.

Phase bump

The audit placed Experiments in Phase 4. This spec moves the A/B to History connection to Phase 2/3 — it is a core value-prop moment, not polish. Build the connection before the visual refinement pass.

A/B flow

1

Load prompts

Two input slots: Prompt A and Prompt B. Each has a "Load from Library" button and a direct paste area. A "Send current prompt to A/B" one-click entry point also lives in the Create output toolbar.

2

Configure run

Model selector (applies to both). Optional rubric (3 to 5 criteria max, inline text fields). Number of runs (1–5). No modal — all inline in a collapsible config strip above the Run button.

3

Run comparison

Single Run button. Results surface side by side. Streaming output shows in both columns simultaneously. No modal churn mid-run.

4

Evaluate and save

Auto-saved (no user action): run metadata — prompt text for A and B, model, timestamp, run count, raw outputs. This always writes to History silently.
Explicit save required: rubric scores, winner selection (A / B / Tie), and outcome notes. These require the user to press "Save outcome." The button is prominent in the result area. History rows without an explicit save are shown with a "Pending evaluation" state — not silently treated as complete outcomes.

History panel

Persistent right panel (320px), collapsible. Shows past experiment runs as a timeline. Each row shows: prompt name pair, winner if declared, date, pending/complete status. Clicking a row loads that run's prompt pair back into A/B slots for re-run or comparison. Directly connected — no separate History tab.

The onboarding step must use the same mental model as Connection Center. Onboarding step 2 becomes: "Connect your provider — Anthropic or Ollama." The current options page at options.html serves as a fallback config surface for users who access it via chrome://extensions. It should present the same Connection Center layout, not a divergent form. Shared component or at minimum shared mental model — same field labels, same order, same terminology. PLB-SPEC-001 · Section 06

Composer

Composer builds multi-block prompt sequences from saved Library items. It is a utility feature, not a primary workspace. Its long-term home is as a secondary action available from both Create and Library — not a product top-level tab, and not buried inside Experiments.

IA resolution

Composer is currently half-integrated — mentioned inside Experiments in the original spec but functionally separate from A/B. Resolved: Composer lives as a utility mode accessible from two entry points. It does not get its own nav tab.

PLB-SPEC-001 · Section 07

Panel governance

The right edge is used by three panels: the Library drawer (from Create), the Save drawer (from Create output), and the Library detail panel (from Library screen). Without a governance rule, these will collide.

Rule: one right-side panel at a time

Only one right-side panel may be open at any moment. Opening a second panel closes the first. There is no stacking, no layering, and no partial overlap. The active panel is always a full-height overlay anchored to the right edge.

Precedence table

TriggerPanel openedIf another panel is open
Library → (from Create editor toolbar)Library drawerCloses Save drawer first, no warning
Save ↗ (from Create output toolbar)Save drawerCloses Library drawer first, no warning (input content is not lost)
Prompt title click (from Library screen)Library detail panelCloses any other panel
Escape keyCloses active panelReturns focus to the trigger element

Pad

Pad does not use the right edge. It opens as a floating overlay (centered, fixed dimensions) or a bottom sheet on narrow viewports. This avoids collision with all right-side panels entirely. Pad is dismissed by Escape or clicking outside it.

Panel widths

PanelWidthNote
Library drawer (from Create)320pxSearch + flat list. Narrower because the Create editor needs to remain visible behind it.
Save drawer320pxForm fields only. Same width as Library drawer for consistent panel feel.
Library detail panel380pxWider because it shows full prompt text, version history, and notes.

Mobile / narrow panel mode

At panel widths below 500px (the extension panel in a narrow split), right-side drawers become bottom sheets that slide up to 60% viewport height. The rule of one panel at a time still applies.

PLB-SPEC-001 · Section 08

Visual direction

Target aesthetic: workstation. Not generic dark utility panel. Not consumer app. A tool that serious prompt engineers would expect to live in alongside their IDE.

Typography

RoleSizeWeightUse
Body14–15px400All primary content. Up from current text-xs reliance.
Label13px500Form labels, card titles, nav items.
Meta12px400Timestamps, tags, secondary info.
Code12px mono400Prompt text, diff output, API references.
Micro11px600 uppercaseSection labels only. Reduce frequency by 70% from current. No micro labels on every group.

Color roles

One accent color (purple-ish, current accent works). All other color is semantic: green for success/connected, amber for warnings/degraded, red for errors/destructive, blue for informational/linked.

Light mode

Light mode is a real mode, not a partial inversion. The global body background in index.css line 4 must not stay dark when the UI theme switches. Fix requires a proper CSS theme class on <body> with full token overrides, not just swapping a few surface colors.

Spacing

Current micro-buttons and dense strips should move to 44px minimum touch target on primary actions. Use 8px grid throughout. Padding inside cards: 16px. Gap between sections: 24–32px. No 4px gaps between functional elements.

Persistent status chips (toolbar)

Provider, model, and connection state visible at all times in the toolbar right. Users should never have to open Settings to know if the extension is ready. Spec in Navigation section (01).

PLB-SPEC-001 · Section 09

Accessibility standards

These are release gates, not a final cleanup bucket. Each phase ships only when the accessibility requirements for that phase pass. See the per-phase gate table below.

Per-phase gate

Accessibility is not deferred to Phase 4. Every phase has its own gate. A phase is not shippable until its gate items are verified. The Phase 4 items are those that require the full visual system to be in place first — they are not catch-up work.

Phase gate assignments

PhaseGate items
Phase 1Status chip communicates state with dot + text (not color only). Connection Center form fields have visible labels. All new nav items have focus states.
Phase 2Save drawer has focus trap and Escape close. Output tabs use ARIA tablist. Editor toolbar buttons have aria-labels. Save / Enhance CTAs meet 44px target size.
Phase 3Library overflow menu is keyboard-navigable with arrow keys. Delete is separated by divider, labeled with object name, requires confirmation. Detail panel has focus trap and Escape close.
Phase 4Full contrast audit against WCAG AA. Focus ring visual audit. Light mode full accessibility pass. Typography scale verified at all sizes.

Target sizes

Rule

Minimum 44×44px touch target for all primary actions in panel mode. Secondary actions (overflow, copy) minimum 32×32px. Current icon-only buttons in the editor toolbar and library cards fall below this — all require remediation. Source: App.jsx:1064

Icon-only buttons

Every icon-only button must have an accessible name via aria-label or title. The name must describe the action, not the icon: "Delete prompt" not "Trash." Current usage at App.jsx:1338 and editor toolbar.

Keyboard completeness

InteractionRequirement
Composer drag-and-dropMove up / Move down keyboard controls on every block
Library card actionsFull keyboard access to overflow menu via arrow keys
A/B prompt slotsTab-accessible load controls, not drag-only
Save drawerFocus trap within drawer when open, Escape closes
Output tabsArrow key navigation between tabs (standard ARIA tablist pattern)

Focus states

Visible focus rings on all interactive elements. Border-color shift alone is insufficient — use a 2px offset ring in the accent color. Source: current index.css reliance on border-color transitions only.

Color contrast

ElementIssueFix
Muted gray metadata text (timestamps, tags)Likely below 4.5:1 on dark surfaceUse text-sm equivalent at #9a9894 minimum on #17171a surface
text-xs uppercase microcopySmall size + low weight + muted color = triple contrast failureRaise to 12px minimum, increase weight to 600, audit color
Placeholder text in inputsPlaceholder-only affordance for required fieldsAdd visible labels above inputs, not just placeholder

Destructive action patterns

Delete must not sit adjacent to high-frequency actions like Copy or Use. In the overflow menu, Delete is always last, always separated by a divider, always labeled with the object name: "Delete this prompt" not "Delete." Confirmation copy must be specific. Source: App.jsx:1338

State communication

Never convey state through color alone. Connection status chips use dot + label (not dot only). Error states in the editor use icon + text (not red border only). Loading states use text ("Enhancing...") in addition to any visual indicator.
